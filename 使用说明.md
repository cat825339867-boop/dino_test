# 使用说明

## 1. 当前方案的核心原则

当前项目已经按职责拆成了独立部分，核心要求是：

- `DINO` 的地方不能出现 `FAISS` 逻辑
- `FAISS` 的地方不能出现 `DINO` 逻辑
- 建库流程必须按“两步走”执行

因此现在的分工是：

- [export_dino_features.py](/c:/Users/leiyu/Desktop/aaaaa/export_dino_features.py)：纯 DINO 环境使用
- [build_val_faiss_index.py](/c:/Users/leiyu/Desktop/aaaaa/build_val_faiss_index.py)：纯 FAISS 环境使用
- [fastapi_dino_service.py](/c:/Users/leiyu/Desktop/aaaaa/fastapi_dino_service.py)：纯 DINO 服务
- [fastapi_faiss_service.py](/c:/Users/leiyu/Desktop/aaaaa/fastapi_faiss_service.py)：纯 FAISS 服务
- [fastapi_box_match_service.py](/c:/Users/leiyu/Desktop/aaaaa/fastapi_box_match_service.py)：编排服务，只调用远程服务，不本地做 DINO 或 FAISS

---

## 2. 现在各脚本分别做什么

### 2.1 `export_dino_features.py`

这是纯 DINO 特征导出脚本。

作用：

1. 扫描图片目录
2. 用 DINOv3 提取特征
3. 导出：
   - `features.npy`
   - `id_to_path.json`

注意：

- 这个脚本里没有 `faiss` 逻辑
- 它只能在装了 DINO / torch / transformers 的环境里运行

---

### 2.2 `build_val_faiss_index.py`

这是纯 FAISS 建库脚本。

作用：

1. 读取 `features.npy`
2. 读取 `id_to_path.json`
3. 建立 `val.faiss`
4. 保存索引输出

注意：

- 这个脚本里没有 `DINO` 逻辑
- 它只能在装了 `faiss` 的环境里运行
- 它不再支持直接从图片目录建库

---

### 2.3 `fastapi_dino_service.py`

这是纯 DINO 服务。

作用：

1. 接收图片路径和框列表
2. 裁剪每个框
3. 提取每个 crop 的 DINO 特征
4. 返回特征向量给上游服务

注意：

- 这里没有任何 FAISS 检索逻辑

---

### 2.4 `fastapi_faiss_service.py`

这是纯 FAISS 服务。

作用：

1. 根据 `label_name` 找到对应标签目录
2. 加载预构建好的：
   - `val.faiss`
   - `id_to_path.json`
3. 对外提供向量检索接口

注意：

- 这里没有任何 DINO 特征提取逻辑
- 这里不会从图片临时建库
- 如果索引文件没准备好，会直接报错

---

### 2.5 `fastapi_box_match_service.py`

这是编排服务。

作用：

1. 接收业务请求
2. 调用 `DINO 服务` 获取每个框的特征
3. 调用 `FAISS 服务` 对每个特征做检索
4. 根据阈值判断哪些框是 `correct` / `wrong`

注意：

- 它自己不直接 import `dino.py`
- 它自己不直接操作 `faiss`
- 它只做远程调用和结果汇总

---

## 3. 建库流程怎么跑

现在建库必须分成两步。

---

### 第一步：在 DINO 环境导出特征

运行命令：

```bash
python export_dino_features.py --val-dir ./shuicao --output-dir ./faiss_label_store/shuicao
```

这一步的含义是：

- 输入目录：`./shuicao`
- 输出目录：`./faiss_label_store/shuicao`

输出结果会是：

```text
faiss_label_store/
  └── shuicao/
      ├── features.npy
      └── id_to_path.json
```

其中：

- `features.npy`：DINO 提取出的特征矩阵
- `id_to_path.json`：向量 ID 到图片路径的映射

---

### 第二步：在 FAISS 环境建立索引

运行命令：

```bash
python build_val_faiss_index.py \
  --feature-path ./faiss_label_store/shuicao/features.npy \
  --mapping-path ./faiss_label_store/shuicao/id_to_path.json \
  --output-dir ./faiss_label_store/shuicao
```

这一步会在同一个目录下补上 `val.faiss`：

```text
faiss_label_store/
  └── shuicao/
      ├── features.npy
      ├── id_to_path.json
      └── val.faiss
```

这就是当前 `FAISS 服务` 需要的标准目录格式。

---

## 4. 最终目录应该长什么样

推荐目录结构如下：

```text
faiss_label_store/
  ├── shuicao/
  │   ├── features.npy
  │   ├── id_to_path.json
  │   └── val.faiss
  ├── faucet/
  │   ├── features.npy
  │   ├── id_to_path.json
  │   └── val.faiss
  └── drain/
      ├── features.npy
      ├── id_to_path.json
      └── val.faiss
```

其中：

- 目录名就是 `label_name`
- `FAISS 服务` 会按这个规则去找索引：
  - `<LABEL_FAISS_ROOT>/<label_name>/val.faiss`
  - `<LABEL_FAISS_ROOT>/<label_name>/id_to_path.json`

例如当请求里传：

```json
{
  "label_name": "shuicao"
}
```

`FAISS 服务` 就会去找：

```text
faiss_label_store/shuicao/val.faiss
faiss_label_store/shuicao/id_to_path.json
```

## 5. 三个服务怎么理解

### 5.1 DINO 服务

启动文件：

- [fastapi_dino_service.py](/c:/Users/leiyu/Desktop/aaaaa/fastapi_dino_service.py)

职责：

- 图片裁剪
- 特征提取

不负责：

---

### 5.2 FAISS 服务

启动文件：

- [fastapi_faiss_service.py](/c:/Users/leiyu/Desktop/aaaaa/fastapi_faiss_service.py)

职责：

- 加载预构建索引
- 接收向量并检索

不负责：

- 图片处理
- DINO 特征提取

---

### 5.3 Box 编排服务

启动文件：

- [fastapi_box_match_service.py](/c:/Users/leiyu/Desktop/aaaaa/fastapi_box_match_service.py)

职责：

1. 接收 `image_path + label_name + boxes`
2. 调 DINO 服务提特征
3. 调 FAISS 服务检索
4. 汇总返回 `correct / wrong`

不负责：

- 本地提 DINO 特征
- 本地操作 FAISS

---

## 6. 服务调用链路

现在完整的调用关系是：

```text
客户端
  -> Box 编排服务
  -> DINO 服务
  -> FAISS 服务
  -> 返回结果
```

更细一点是：

```text
image_path + label_name + boxes
    -> Box 编排服务
    -> 调 DINO 服务提每个框的特征
    -> 调 FAISS 服务检索对应标签库
    -> 用 score 和 threshold 比较
    -> 返回每个框是 correct / wrong
```

## 8. 环境变量说明

### 8.1 Box 编排服务

可以通过环境变量覆盖远程服务地址：

- `DINO_SERVICE_URL`
- `FAISS_SERVICE_URL`

例如：

```bash
export DINO_SERVICE_URL=http://127.0.0.1:8002
export FAISS_SERVICE_URL=http://127.0.0.1:8001
python fastapi_box_match_service.py
```

---

### 8.2 FAISS 服务

可以通过环境变量指定索引根目录：

- `LABEL_FAISS_ROOT`

例如：

```bash
export LABEL_FAISS_ROOT=./faiss_label_store
python fastapi_faiss_service.py
```

## 9. Box 编排服务请求示例

请求体示例：

```json
{
  "image_path": "D:/data/test.png",
  "label_name": "shuicao",
  "boxes": [
    { "x1": 10, "y1": 20, "x2": 200, "y2": 220 },
    { "x1": 240, "y1": 30, "x2": 420, "y2": 260 }
  ],
  "threshold": 0.65,
  "topk": 1
}
```

返回逻辑：

1. Box 服务调 DINO 服务提取每个框的特征
2. Box 服务调 FAISS 服务检索 `shuicao` 标签库
3. 根据返回的 `top1 score` 和 `threshold` 比较
4. 返回每个框是：
   - `correct`
   - `wrong`
   - `error`

## 10. 需要特别注意的点

### 10.1 `build_val_faiss_index.py` 不能再直接传图片目录

现在这个脚本已经改成纯 FAISS 脚本，因此它不再支持：

```bash
python build_val_faiss_index.py --val-dir ./shuicao --output-dir ./faiss_label_store
```

以后必须先在 DINO 环境导出特征，再在 FAISS 环境建索引。

---

### 10.2 `FAISS 服务` 只认预构建索引

现在 `FAISS 服务` 不会再：

- 从图片目录临时提特征
- 自动建库

所以如果目录里没有：

- `val.faiss`
- `id_to_path.json`

它就会直接报错。

---

### 10.3 Linux 下不要写 Windows 风格路径

在 Linux / Ubuntu 下运行脚本时，当前目录应该写：

```bash
python ./build_val_faiss_index.py
```

不要写：

```bash
python .\build_val_faiss_index.py
```

因为：

- `.\xxx.py` 是 Windows 风格
- `./xxx.py` 才是 Linux 风格
